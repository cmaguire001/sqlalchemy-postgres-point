// 01 — overview

How it works

Four validation layers fire in order on every coordinate write or read.

STEP 01

Shape & finiteness

Must be a 2-tuple of finite floats. No NaN, no Infinity, no 3D points. Fails immediately with a clear ValueError before any range check runs.

STEP 02

Range check

lng ∈ [−180, 180] and lat ∈ [−90, 90]. Hard out-of-range values are flagged INVALID. These cannot be salvaged.

STEP 03

Swap detection

First value in lat range but second in lng-only range (90, 180]? Flagged SWAPPED — not invalid. Real data, wrong order. Separate bucket.

STEP 04

Ambiguity check

Both values in [−90, 90]? Either could be latitude. In strict mode, these surface for human review rather than silently passing.

// 05 — failure modes

All the ways GPS data goes wrong

Six distinct failure patterns — each requires a different response from your pipeline.

PATTERN 01 — INVALID

Hard out-of-range

Longitude > 180 or latitude > 90. No valid interpretation exists. Common causes: unit confusion (radians vs degrees) or sensor firmware bugs. Quarantine and escalate.

(181.0, 0.0) → ValueError: Longitude must be within [−180, 180]
PATTERN 02 — SWAPPED

Transposed (lat, lng)

The most common real-world mistake. Especially prevalent for cities where longitude clearly exceeds 90: Tokyo (139°), Sydney (151°), Singapore (103°). The data is valid — the order is wrong. Auto-correctable.

(35.68, 139.69) → SWAPPED — fix: (139.69, 35.68)
PATTERN 03 — AMBIGUOUS

Both values in lat range

Points in Europe, Africa, or central Asia where both values fall in [−90, 90]. Mathematically valid as either order. Flag for human review — cross-reference with expected geographic region.

(45.0, 60.0) → AMBIGUOUS — cannot confirm axis order
PATTERN 04 — SCIENTIFIC

Scientific notation from PostgreSQL

Coordinates near (0,0) may come back from PostgreSQL as 1e-10. A naive regex silently fails to parse these. Fix: extend the capture group to (-?[\d.]+(?:[eE][+-]?\d+)?). Built into this library.

"(1e-10,2e-10)" → parsed as (1e-10, 2e-10) ✓
PATTERN 05 — NaN / INFINITY

Non-finite floats

Float overflow, missing sensor data, or divide-by-zero upstream produces NaN or inf. Caught by math.isfinite() before any range check — fails fast with an unambiguous error message.

(float('nan'), 40.0) → ValueError: non-finite coordinate
PATTERN 06 — WRONG SHAPE

3-tuple, scalar, or string

3D GPS data (x, y, z), a bare float, or a string passed by mistake. Length is validated before everything else — no silent truncation, no index errors buried ten layers downstream.

(-74.0, 40.7, 0.0) → ValueError: expected 2-length tuple
Classification decision tree
1. 2-tuple of finite floats? → No → ValueError
2. Either value strictly OOB?
→ |lng| ≤ 90 and 90 < |lat| ≤ 180 → SWAPPED
→ Otherwise → INVALID
3. Both values in [−90, 90]? → AMBIGUOUS
4. |lng| > 90 and lat in range? → VALID
Key principle: swapped ≠ invalid. Swap means fixable with a reorder. Invalid means the data itself is corrupt. Handle them differently — auto-correct swaps, quarantine invalids, flag ambiguous for review.
