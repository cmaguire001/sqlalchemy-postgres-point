How it works

STEP 01

Define your model

Add PointType() to any SQLAlchemy column. Pass a (lng, lat) tuple — the type handles the rest.

STEP 02

Validation on write

On bind, coordinates are range-checked. Out-of-bounds values raise ValueError immediately — no silent corruption.

STEP 03

Swap detection

When a likely (lat, lng) is passed instead of (lng, lat), the type flags it separately from hard-invalid data.

STEP 04

Batch analysis

Run validate_points() on a list of rows to get a summary report: swap %, invalid %, ambiguous %, and flagged rows.

// 02 — code

Usage

models.py from sqlalchemy import Column, Integer from sqlalchemy_postgres_point import PointType class Location(Base): __tablename__ = "locations" id = Column(Integer, primary_key=True) coords = Column(PointType()) # stores as PostgreSQL POINT # Write — pass (lng, lat) location = Location(coords=(-120.0, 40.0)) # ✓ valid location = Location(coords=(40.0, -120.0)) # ⚠ flagged as likely swapped location = Location(coords=(181.0, 0.0)) # ✗ raises ValueError
validation.py from sqlalchemy_postgres_point import validate_points summary = validate_points([ (1.0, 2.0), # ✓ valid (40.0, -120.0), # ⚠ likely swapped (lat, lng) (181.0, 0.0), # ✗ invalid longitude (50.0, 20.0), # ~ ambiguous (both in lat range) ]) # summary["invalid_pct"] → 25.0 (swapped ≠ invalid) # summary["likely_swapped_pct"] → 25.0 # summary["ambiguous_pct"] → 25.0

What each status means

VALID lng ∈ [−180,180], lat ∈ [−90,90]
INVALID coordinate strictly out of range
SWAPPED looks like (lat,lng) was passed instead of (lng,lat) — flagged separately from invalid
AMBIGUOUS both values fall in lat range; order uncertain
Key insight:
A swapped point is NOT the same as an invalid point. (40, −120) is real GPS data — just written in the wrong order. The library tracks these separately so your data pipeline can handle them differently.
